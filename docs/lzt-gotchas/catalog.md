# Грабли API: категории, эндпоинты, поиск

## Слаг категории — не название метода поиска

Ловушка: если бы имя метода строилось по правилу "взять слаг, добавить
`category_`" — оно было бы неверным для пяти категорий из общего списка.

`epicgames` → `category_epic_games`, `tiktok` → `category_tik_tok`,
`battlenet` → `category_battle_net`, `escapefromtarkov` →
`category_escape_from_tarkov`, `socialclub` → `category_social_club`.

Механическое построение имени по слагу не работает — соответствие слаг →
эндпоинт нужно держать явным списком, а не выводить по шаблону.

Источник: `projects/flow/app/domain/market/adapter.py` (комментарий над
`_CATEGORY_METHODS`).

## Пять категорий видны в каталоге, но у них нет поиска

Ловушка: категория присутствует в общем списке категорий маркета — не
значит, что по ней можно искать.

`mihoyo`, `wot`, `wotblitz`, `vkontakte`, `other` — это категории, которые
маркет показывает и продаёт, но под них **нет** отдельного
`category_*`-эндпоинта поиска вообще. Попытка вызвать поиск по одной из
них упадёт не валидацией параметров, а отсутствием самого метода.

Источник: `projects/flow/app/domain/market/categories.py:29-31`.

## У каждой категории — свой набор фильтров, и размер сильно разный

Ловушка: "раз это один и тот же маркет — фильтры должны быть примерно
одинаковые" не работает.

`category_steam` принимает **порядка 125 параметров** поиска (строки
3764–3892 файла фасада). `category_vpn` — **около 25** (строки
4972–4998). Разница на порядок, и оба варианта — норма для своей
категории, не недосмотр.

Источник: `projects/pylzt/src/pylzt/facades/market.py:3764-3892` (steam),
`:4972-4998` (vpn).

## Одинаковые параметры в разных категориях идут в разном порядке

Ловушка: если один раз запомнил порядок параметров одной категории —
он не переносится на другую, даже когда имена параметров совпадают.

В `category_steam` порядок — `email_type`, затем `email_provider`
(строки 3784–3785). В `category_fortnite` — наоборот: `email_provider`,
затем `email_type` (строки 2368–2369). Отсюда правило: параметры поиска
по категории — только по имени (keyword), никогда позиционно.

Источник: `projects/pylzt/src/pylzt/facades/market.py:2368-2369` (fortnite),
`:3784-3785` (steam).

## Спецификация батча не описывает собственное тело запроса

Ловушка: у `POST /batch` в официальной OpenAPI-спеке "массив job-объектов"
как форма тела — но это не плоский набор полей, и обычный
кодогенератор такую форму не видит вообще: он получает эндпоинт без
единого поля запроса.

Хуже: спека также **угадывает** форму ответа (`Jobs`/`JobsJobId`) — и
угаданная форма не совпадает с реальной. Живой ответ сервера — это
динамическая map по job-id, а не структура, которую описывает спека.

Источник: `projects/pylzt/src/pylzt/methods/market_batch_requests.py`,
`projects/pylzt/src/pylzt/methods/forum_batch_requests.py`.
