# Контрибуция в open-lzt

Спасибо, что помогаешь. Есть **три разных способа** контрибьютить, и у них разные правила, потому что
они несут разный уровень доверия. Найди свой ниже.

Новичок в экосистеме? Сперва прочитай [docs/WHY.md](docs/WHY.md) и
[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) — они объясняют, как всё складывается, и после этого
остальное становится очевидным.

---

## Выбери свою поверхность контрибуции

| Я хочу… | Уровень доверия | Где | Гайд |
|---|---|---|---|
| **Поделиться автоматизацией** (флоу, как данные) | Кто угодно | каталог / API flow | [flow-design-guide](https://github.com/open-lzt/auto-lzt/blob/main/docs/flow-design-guide.md) · [modules](https://github.com/open-lzt/auto-lzt/blob/main/docs/modules.md) |
| **Добавить новый узел/фичу** (Python-код) | Только владелец | плагин flow | [plugins](https://github.com/open-lzt/auto-lzt/blob/main/docs/plugins.md) |
| **Улучшить core-проект** (SDK, движок, багфикс) | Проверяемый PR | любой репо | этот файл, §Core-PR |

---

## 1. Поделиться флоу (без кода, кто угодно)

**FLOW-модуль** (`kind: flow`) — это автоматизация, выраженная как *данные*: граф
`триггер → выборка → фильтр → действие → уведомление`. Поскольку это данные, публиковать может кто
угодно, а валидатор проверяет их перед запуском.

- Спроектируй его по [гайду дизайна флоу](https://github.com/open-lzt/auto-lzt/blob/main/docs/flow-design-guide.md).
- Или опиши простым текстом и дай помощнику `flow-from-text` набросать и провалидировать FlowSpec.
- Публикация и каталог описаны в [modules.md](https://github.com/open-lzt/auto-lzt/blob/main/docs/modules.md).

Это самая дружелюбная точка входа — Python не нужен.

## 2. Написать Python-плагин (код, только владелец)

**Python-плагин** (`kind: python`) добавляет реальную исполняемую возможность — новые типы узлов,
роутеры, хендлеры. **Это доверенный код без песочницы**: по определению он может делать что угодно с
аккаунтом. Именно поэтому плагины — **только для владельца**, ставятся через `pip` + перезапуск сервиса,
*никогда* не принимаются через API от незнакомца. Запуск произвольного кода от недоверенных авторов был
бы дырой в безопасности, а не фичей.

- Следуй [plugins.md](https://github.com/open-lzt/auto-lzt/blob/main/docs/plugins.md): минимальная
  структура — точка входа в `pyproject.toml` + `nodes.py` с паттерном `REGISTRATIONS`.
- Если предлагаешь плагин для основного проекта, сперва открой issue — он будет ревьюиться как
  доверенный код, по тем же стандартам, что и §3.

## 3. Улучшить core-проект (проверяемый PR)

Багфиксы и фичи в `pylzt`, `testnet`, `eventus`, `eventus-sdk`, `flow` или `mcp` идут через обычный
pull request.

**Вайбкод-PR принимаются** — писал ты код сам или с ИИ, не важно. Важен результат. Но:

- **Небольшой.** Одно понятное изменение, а не «ИИ нагенерил 2000 строк по всему проекту». Большой
  рефактор — сперва issue.
- **Не говнокод.** Типизация (`mypy --strict`, DTO на границах, никакого `Any`), слоение
  (Handler → Service → Repo, без логики в хендлере и SQL вне репозитория), тесты на новое поведение.
- **С подробным описанием.** Что и зачем меняешь; если это багфикс — воспроизведение и в чём была
  причина, а не только симптом.

Прогони чек-лист ниже перед тем как открыть PR — он и есть граница между «вайбкод» и «говнокод».

### Настройка окружения

Каждый проект — самостоятельный пакет [`uv`](https://docs.astral.sh/uv/). Работай внутри того, что
меняешь:

```bash
cd projects/<name>
uv sync --all-extras
uv run pytest -q                       # прогнать сьют
```

Стенд по умолчанию в **testnet**, и тесты тоже — они гоняются против мок-маркета, токен не нужен.
Только opt-in маркеры (`-m live`, `-m pg`) трогают реальную инфру; не включай их без надобности.

### Перед тем как открыть PR — чек-лист

```bash
cd projects/<name>
uv run ruff format --check .
uv run ruff check .
uv run mypy src        # 'app' для flow
uv run pytest -q       # зелёный, включая дефолтный e2e где он есть
```

- **Типизировано.** `mypy --strict` чистый. Никакого `Any` для срезания углов, никакого «голого»
  `dict` через границу — DTO.
- **Покрыто тестом.** Новое поведение получает тест. Багфикс получает регресс-тест. Сьют остаётся зелёным.
- **Чисто по линту.** `ruff` + `ruff format` проходят.
- **Безопасно для testnet.** Ни один тест по умолчанию не бьёт в реальный маркет.
- **Сфокусировано.** Одно логическое изменение на PR. Обнови доки затронутого проекта (`docs/`, `_MODULE.md`).
- **Без секретов.** Никогда не коммить `.env`, `*.session`, `*.db`, токены или ключи.

Сообщения коммитов описывают **видимое пользователю изменение**, а не внутренний механизм. Один коммит
на одно логическое изменение.

## Репорт багов и безопасность

- **Баги / фичи:** открой issue в соответствующем репо. Приложи шаги воспроизведения, проект и был ли
  ты на testnet или prod.
- **Безопасность:** **не** открывай публичный issue на уязвимость — свяжись с мейнтейнером
  ([@zlexdev](https://github.com/zlexdev)) приватно. Учитывая модель доверия плагинов, считай всё,
  что касается owner-only пути кода, чувствительным.

---

Контрибуча, ты соглашаешься, что твоя работа выпускается под [лицензией MIT](LICENSE) проекта.
Автоматизируй с умом и на своих аккаунтах.
