# open-lzt — Архитектура

> Один документ — вся экосистема: каждый репозиторий, как они связаны и как разворачивается стенд.
> Написан так, чтобы его можно было прочитать от начала до конца — ИИ-агенту или новому
> контрибьютору. В каждом репозитории вдобавок лежит папка `docs/for_ai/` с машинно-ориентированной
> картой его внутренностей — этот файл на уровень *выше*, он описывает швы **между** репозиториями.

---

## 1. Модель в одном абзаце

`pylzt` — типизированный async-SDK, который говорит на API lzt.market / lolzteam. `testnet` — мок-сервер,
который говорит на *том же самом* API, поэтому всё, что построено на `pylzt`, может работать оффлайн.
`eventus` опрашивает маркет через `pylzt`, превращает изменения в долговечный, воспроизводимый **лог
событий** и доставляет их по REST / webhook / SSE / WS; `eventus-sdk` — его клиент. `flow` — no-code
движок автоматизаций: он исполняет заданные пользователем **граф-флоу** (триггер → выборка → фильтр →
действие → уведомление), используя `pylzt` для действий. `mcp` отдаёт всё это ИИ-агенту в виде
инструментов. Монорепо `open-lzt` собирает пять запускаемых сервисов в единый `systemd`-стенд, по
умолчанию в testnet, так что ничего не касается реального аккаунта, пока ты не переключишь `MARKET_MODE`.

## 2. Слои стека

```
                            ┌───────────────────────────────────────────┐
   ИИ-агент / MCP-клиент ──▶│  mcp            (:8770)  управление + тест │
                            └───────────────────────────────────────────┘
                                     │ инструменты зовут ▼
        ┌────────────────────────────┴───────────────┬──────────────────────────┐
        ▼                                             ▼                          ▼
┌───────────────────┐                       ┌───────────────────┐      ┌───────────────────┐
│ flow   (:8000 api │                       │ eventus  (:27543) │◀────▶│ eventus-sdk       │
│        + worker)  │                       │ poll→log→delivery │      │ (клиентская либа) │
│ no-code автоматиз.│                       └─────────┬─────────┘      └───────────────────┘
└─────────┬─────────┘                                 │
          │  оба ходят в маркет через                 │
          ▼            ОДИН И ТОТ ЖЕ SDK ◀─────────────┘
                 ┌───────────────────────────┐
                 │ pylzt   (SDK, библиотека) │  пул токенов · рейт-лимит · прокси · ретраи
                 └─────────────┬─────────────┘
                               │ HTTP, base URL выбирается по MARKET_MODE
              ┌────────────────┴─────────────────┐
              ▼ testnet                            ▼ prod
     ┌───────────────────┐               ┌───────────────────┐
     │ testnet  (:8765)  │               │ api.lzt.market    │
     │ мок-маркет        │               │ (реальный маркет) │
     └───────────────────┘               └───────────────────┘
```

**Направление зависимостей строго вниз.** `flow`, `eventus` и `mcp` зависят от `pylzt`; от
`flow`/`eventus`/`mcp` не зависит ничего, кроме `mcp` (который оркестрирует остальные два) и
`eventus-sdk` (тонкий клиент `eventus`). `pylzt` не зависит ни от чего в этой экосистеме.

## 3. Репозиторий за репозиторием

### pylzt — `github.com/open-lzt/pylzt`
Фундамент. Типизированный async-клиент, сгенерированный из OpenAPI-спеки lzt.market, плюс API форума
lolzteam и AntiPublic. Это не тонкая обёртка — он владеет **пулом токенов** (round-robin по множеству
токенов аккаунтов), **рейт-лимитом**, **ротацией прокси**, пагинацией, загрузкой медиа и типизированной
обработкой ошибок. Каждый остальной сервис ходит в маркет *только* через этот SDK — именно поэтому
подмена base URL (testnet ↔ prod) это правка в одну строку, о которой больше никому знать не надо.

- **Зависит от:** ничего внутри экосистемы.
- **Используется:** `eventus`, `flow`, `mcp` и напрямую пользовательскими скриптами.

### testnet — `github.com/open-lzt/lzt-testnet`
FastAPI-сервер, воспроизводящий поверхность API lzt.market — те же роуты, те же формы ответов — на
in-memory состоянии. Он существует, чтобы весь стек можно было запускать и тестировать **без реального
токена и без реального маркета**. Его эндпоинты (`/testnet/reset`, `/testnet/revoke-token`) позволяют
тестам детерминированно проигрывать граничные случаи. В стенде слушает `:8765`.

- **Зависит от:** ничего (это то, от чего зависят другие в testnet-режиме).
- **Используется:** каждым сервисом, когда `MARKET_MODE=testnet`.

### eventus — `github.com/open-lzt/lzt-eventus`
Движок событий. Опрашивает маркет через `pylzt`, раскладывает состояние в **доменные события**
(например, новый заказ, проданный товар), пишет их в долговечный, **воспроизводимый** лог и доставляет
через catch-up-шину по REST, webhook (с HMAC-подписью), SSE и WebSocket. Состояние живёт в Postgres-базе
`lzteventus`. В стенде слушает `:27543` (`/healthz`, `/subscriptions`).

- **Зависит от:** `pylzt`, Postgres, Redis.
- **Используется:** `eventus-sdk`, `mcp` и (опционально, встроенно) воркером `flow`. Точки расширения
  описаны в его `docs/extending.md`.

### eventus-sdk — `github.com/open-lzt/lzt-eventus-sdk`
Клиентская async-библиотека к management-API `eventus`: создать/получить подписки, читать поток событий,
проверять webhook-подписи. Именно её импортирует внешнее приложение, чтобы *потреблять* события, не
переизобретая протокол доставки.

- **Зависит от:** `eventus` (как запущенный сервис, по HTTP).
- **Используется:** внешними приложениями-потребителями.

### flow — `github.com/open-lzt/auto-lzt`
No-code движок автоматизаций и самый большой сервис. **Флоу** — это JSON-граф узлов
(`триггер → выборка → фильтр → действие → уведомление`); движок его компилирует, а воркер исполняет,
используя `pylzt` для действий на маркете. Работает как **два юнита**: HTTP-API (`:8000` — CRUD флоу,
каталог, запуски) и воркер (очередь задач arq + APScheduler для временных триггеров + опционально
встроенный роутер eventus). Состояние живёт в Postgres-базе `lztflow`. Конфиг использует префикс
`LZT_FLOW_`. Расширяется двумя способами — см. §6.

- **Зависит от:** `pylzt`, `eventus` (как библиотека, `lzt-eventus[engine]`), Postgres, Redis.
- **Используется:** `mcp`, Telegram-ботом администратора (опционально) и конечными пользователями через API.
- **Каноничные гайды:** `docs/flow-design-guide.md`, `docs/modules.md`, `docs/plugins.md`.

### mcp — `github.com/open-lzt/lzt-mcp`
MCP-сервер (Model Context Protocol), дающий ИИ-агенту безопасный типизированный инструментарий над всем
стендом: тестирование сырых API-запросов, управление флоу, интроспекция eventus. Он **по умолчанию
testnet и защищён от prod** — агент не может случайно попасть в реальный маркет. Слушает `:8770`.

- **Зависит от:** `pylzt`, `testnet`, `flow` + `eventus` (как запущенные сервисы), Postgres/Redis через них.
- **Используется:** любым MCP-клиентом (Claude, IDE-агенты и т.д.).

## 4. Как они связаны в рантайме

**Доступ к маркету (все сервисы).** Ни один сервис не строит HTTP-запросы к маркету сам. Они создают
клиент `pylzt` с base URL, выведенным из `MARKET_MODE`. В режиме `testnet` этот URL — локальный сервис
`testnet`; в режиме `prod` — `api.lzt.market`. Это единственный шов, который делает весь стенд безопасным
по умолчанию.

**События (eventus → потребители).** `eventus` — единственный производитель лога событий. Потребители
никогда не опрашивают маркет на изменения сами — они подписываются на `eventus` (через `eventus-sdk`,
webhook или поток SSE/WS) и получают воспроизводимые события. Это держит опрос и давление рейт-лимитов
в одном месте.

**flow ↔ eventus.** Воркер `flow` может *встроить* роутер событий eventus в свой процесс (чтобы событие
могло запускать флоу) — это управляется флагом `LZT_FLOW_EMBED_EVENTUS`. **В собранном стенде это
выключено (`=0`)**: eventus работает как свой отдельный сервис (`open-lzt-eventus`), а воркер flow — это
только arq + планировщик. Когда `flow` запускается *отдельно* (вне стенда), встроенный роутер можно
включить. Две формы деплоя, один кодовый база.

**mcp → всё остальное.** `mcp` — слой оркестрации для агентов. Его инструменты зовут `pylzt` для сырых
запросов, API `flow` для управления автоматизациями и `eventus` для интроспекции событий — всегда
соблюдая testnet-защиту.

## 5. Топология деплоя (стенд `open-lzt`)

Монорепо — это **инфра в Docker, сервисы под systemd**. `docker-compose.yml` поднимает только Postgres
и Redis (привязаны к loopback); пять Python-сервисов работают как `systemd`-юниты через `uv run`.
`install.sh` (идемпотентный) генерирует все секреты, создаёт обе базы, применяет обе цепочки миграций
Alembic и запускает юниты.

| Юнит / контейнер | systemd-юнит | Порт (127.0.0.1) | Хранилище |
|---|---|---|---|
| testnet | `open-lzt-testnet` | `8765` | in-memory |
| eventus | `open-lzt-eventus` | `27543` | Postgres `lzteventus` |
| flow API | `open-lzt-flow-api` | `8000` | Postgres `lztflow` |
| flow worker | `open-lzt-flow-worker` | — | Postgres `lztflow` + Redis |
| mcp | `open-lzt-mcp` | `8770` | через flow/eventus |
| Postgres | docker `open-lzt-postgres-1` | `55432` | тома: `lztflow`, `lzteventus` |
| Redis | docker `open-lzt-redis-1` | `56379` | очереди, дедуп, кэш |
| панель | — (статика в nginx) | `80`/`443` | нет — SPA, всё через `/api/` |

- **Префиксы конфига:** `flow` читает `LZT_FLOW_*`; `eventus` читает `LZT_*`. Верхнеуровневый `.env` —
  единственный источник; `install.sh` рендерит из него по-сервисные `deploy/env/*.env`.
- **Драйверы Postgres:** асинхронный доступ через `asyncpg` (`postgresql+asyncpg://`); jobstore
  APScheduler, который работает только синхронно, — через `psycopg` v3 (`postgresql+psycopg://`).
  **`psycopg2` не используется никогда** — «голый» URL `postgresql://` нормализуется до нужного драйвера
  в коде до создания движка.
- **Режим маркета:** `MARKET_MODE=testnet|prod` в `.env`; переключение в prod дополнительно требует
  реальных токенов в `EVENTUS_TOKENS`. Перезапусти `install.sh`, чтобы применить (он перерендерит env
  и перезапустит).
- **Порты только loopback**; доступ через SSH-туннель либо экспозиция flow/eventus по HTTPS через
  опциональную связку nginx + Let's Encrypt (`deploy/setup_tls.sh`).
- **Панель — не сервис.** `install.sh` собирает её (`pnpm build`) в `projects/flow/frontend/dist`,
  nginx отдаёт эти файлы с диска и проксирует `/api/` на flow. Единственное новое требование к
  хосту — `node` + `pnpm`; без них установка проходит, панель просто не собирается.
- **SSE за прокси.** Блоки потоков (`deploy/nginx/panel.conf`) выключают `proxy_buffering`. С
  буферизацией nginx держит ответ, пока не заполнится буфер, — а поток событий его не заполняет и
  не закрывает, поэтому браузер получает 200 и дальше тишину. Проверяется тестом, который поднимает
  настоящий nginx и требует, чтобы кадр дошёл: `projects/flow/tests/e2e/test_sse_through_nginx.py`.

Как устроена панель — отдельная глава: [panel-architecture.md](panel-architecture.md).

Поверхность эксплуатации: `scripts/healthcheck.sh`, `scripts/smoke.sh`, `update.sh` (rolling-обновление
с health-гейтом), `deploy/autoupdate.sh` (опционально, выключено по умолчанию — см. `docs/AUTOUPDATE.md`).

## 6. Две модели расширения (важно — это не одно и то же)

`flow` расширяется двумя принципиально разными способами. Их путаница — самая частая ошибка.

| | **FLOW-модуль** (`kind: flow`) | **PYTHON-плагин** (`kind: python`) |
|---|---|---|
| **Что это** | Граф-флоу как **данные** (JSON/YAML) | **Исполняемый Python** — новые типы узлов, роутеры, хендлеры |
| **Кто может публиковать** | Кто угодно | **Только владелец** |
| **Доверие** | Недоверенное — проходит через валидатор / CI | **Полностью доверенное** — без песочницы, по определению может делать что угодно с аккаунтом |
| **Как ставится** | Публикуется через API / каталог | `pip install` + перезапуск сервиса — **не** через API |
| **Почему** | Данные можно проверить; код от незнакомцев безопасно принять нельзя (это был бы RCE как фича) | Владелец — единственная доверенная сторона; API-ключ — ключ бота |
| **Гайд** | `docs/modules.md`, `docs/flow-design-guide.md` | `docs/plugins.md` |

Правило большого пальца: **если это данные — публиковать может кто угодно и охраняет валидатор; если это
код — ставит только владелец, и песочницы нет — это осознанный выбор дизайна, а не дыра.**
